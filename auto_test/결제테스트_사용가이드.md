# MOBILPAY 결제 테스트 도구 사용 가이드

> 작성일: 2026-04-05

---

## 개요

가맹점 개발자 입장에서 MOBILPAY REST API 결제연동을 실제로 테스트하는 도구입니다.
결제수단별 다양한 케이스의 테스트 페이지가 제공되며, 결제 완료 후 결과를 자동 수집·분석합니다.

---

## 1단계: 설정

### 1-1. 인증정보 입력

`auto_test/config.json` 파일을 열어서 실제 가맹점 정보를 입력합니다.

```json
{
  "mobilpay": {
    "test_host": "https://test.mobilians.co.kr",
    "prod_host": "https://mup.mobilians.co.kr",
    "use_test": true,
    "MC": {
      "sid": "휴대폰결제_SID",
      "skey": "휴대폰결제_SKEY"
    },
    "CN": {
      "sid": "신용카드_SID",
      "skey": "신용카드_SKEY"
    }
  },
  "server": {
    "port": 3100,
    "ngrok_url": ""
  }
}
```

| 항목 | 설명 |
|------|------|
| `MC.sid` / `MC.skey` | 휴대폰결제용 가맹점 코드 및 서비스 키 |
| `CN.sid` / `CN.skey` | 신용카드용 가맹점 코드 및 서비스 키 |
| `use_test` | `true`: 테스트서버 / `false`: 운영서버 (단, CN은 항상 운영서버) |
| `port` | 로컬 서버 포트 (기본 3100) |
| `ngrok_url` | ngrok 터널링 URL (2단계에서 설정) |

> **참고**: 결제수단별로 SID/SKEY가 다르므로 각각 별도 입력이 필요합니다. 향후 RA(실계좌이체), VA(가상계좌) 추가 시에도 동일한 형식으로 추가하면 됩니다.

### 1-2. 의존성 설치 (최초 1회)

```bash
cd auto_test
npm install
```

## 2단계: ngrok 터널링 설정

ok_url/noti_url 콜백을 KG파이낸셜 서버에서 수신하려면 외부 접근 가능한 URL이 필요합니다.

### 2-1. ngrok 실행

```bash
ngrok http 3100
```

### 2-2. URL 복사

ngrok이 출력하는 Forwarding URL을 복사합니다.
```
Forwarding  https://xxxx-xxx-xxx.ngrok-free.app -> http://localhost:3100
```

### 2-3. config.json에 입력

```json
{
  "server": {
    "port": 3100,
    "ngrok_url": "https://xxxx-xxx-xxx.ngrok-free.app"
  }
}
```

> **주의**: ngrok을 재시작하면 URL이 변경되므로 매번 config.json을 업데이트해야 합니다.

## 4단계: 결제 테스트 진행

### 4-1. 테스트 대시보드 열기

브라우저에서 **http://localhost:3100** 접속

### 4-2. 테스트 케이스 확인

#### 휴대폰결제 (MC) — 6개 케이스

| 케이스 ID | 이름 | 플로우 | 호출방식 | 콜백 |
|----------|------|--------|---------|------|
| MC-01 | 일반결제 + Popup + ok_url만 | 일반 | Popup | ok_url |
| MC-02 | 일반결제 + Popup + ok_url + noti_url | 일반 | Popup | ok_url + noti_url |
| MC-03 | 하이브리드결제 + Popup + ok_url만 | 하이브리드 | Popup | ok_url |
| MC-04 | 하이브리드결제 + Popup + ok_url + noti_url | 하이브리드 | Popup | ok_url + noti_url |
| MC-05 | 일반결제 + Self(페이지전환) + ok_url | 일반 | Self | ok_url |
| MC-06 | 일반결제 + iframe + ok_url | 일반 | iframe | ok_url |

#### 신용카드 (CN) — 10개 케이스

> - config.json의 `use_test` 설정과 **무관하게** 항상 운영서버를 사용합니다.
> - **실제 결제가 발생**하므로, 테스트 후 반드시 결제취소를 진행해주세요.

**기본 플로우 (6개)**

### 4-3. 테스트 실행

1. 원하는 케이스의 **"테스트 시작"** 버튼 클릭
2. 서버가 자동으로 거래등록 API를 호출합니다
3. 거래등록 성공 시 KG파이낸셜 결제창이 자동으로 열립니다
   - Popup: 별도 팝업 창
   - Self: 새 탭
   - iframe: 카드 하단에 임베드
4. 결제창에서 **실제 결제를 진행**합니다 (테스트서버 기준 1,000원)
5. 결제 완료 후 결과가 자동 수집됩니다
6. 카드에 성공(✓) 또는 실패(✗) 상태가 표시됩니다

### 4-4. 케이스별 테스트 포인트

**MC-01, MC-02 (일반결제)**
- 결제창에서 인증+승인이 동시에 처리됩니다
- ok_url로 최종 결과(sign_date 포함)가 수신되는지 확인

**MC-03, MC-04 (하이브리드결제)**
- 결제창에서 인증만 완료됩니다
- ok_url로 인증 결과 수신 → 서버가 자동으로 승인 API 호출
- 승인까지 완료되어야 최종 성공

**MC-02, MC-04 (noti_url 포함)**
- ok_url 외에 noti_url 웹훅도 수신되는지 확인
- noti_url은 최대 20회 재시도하므로 약간 지연될 수 있음

**MC-05 (Self), MC-06 (iframe)**
- 결제창 호출 방식이 다른 것을 확인
- close_url이 자동 설정되어 사용자 취소 시에도 결과 기록

**CN-01, CN-02 (일시불 일반결제)**
- 신용카드 일시불 기본 플로우. 결제창에서 카드 선택 → 인증+승인 동시 처리
- ok_url로 `cn_card_code`, `cn_card_name`, `cn_appr_no` 등 카드 응답 필드 수신 확인
- 결제창 사이즈: 820 x 600 (휴대폰보다 큼)

**CN-03, CN-04 (하이브리드결제)**
- 결제창에서 인증만 완료 → 서버에서 승인 API 자동 호출
- MC와 달리 `mc_user_key`가 불필요

**CN-05 (Self), CN-06 (iframe)**
- MC와 동일한 호출 방식 차이 확인
- 신용카드 결제창 크기 820x600 적용 확인

**CN-07 (할부)**
- 결제창에서 할부 개월이 3개월로 고정되는지 확인
- 금액이 50,000원 (할부 최소금액 충족)
- 응답의 `cn_installment` 값이 "03"인지 확인

**CN-08 (카드사 고정)**
- 결제창에서 신한카드만 노출되는지 확인
- 다른 카드사 선택 불가 확인

**CN-09 (면세 거래)**
- 전액 면세 처리 (`cn_bill_type=10`, `tax_free=1000`)
- 거래등록 요청에 면세 금액 필드 포함 확인

**CN-10 (복합과세)**
- 과세+면세 혼합 처리 (`cn_bill_type=20`)
- `supply_value=455`, `tax=45`, `tax_free=500` 분리 전달 확인

## 5단계: 결과 분석

### 5-1. 결과 분석 대시보드 열기

- 대시보드 우측 상단 **"결과 분석 대시보드 →"** 클릭
- 또는 브라우저에서 **http://localhost:3100/results** 접속

### 5-2. 필터링

- **테스트 케이스**: 특정 케이스만 필터링
- **결과**: 성공/실패/대기중 필터링

### 5-3. 상세 보기

결과 행을 클릭하면 상세 분석이 펼쳐집니다:

**종합 분석 카드**
- ✓ 거래등록 성공 여부
- ✓ ok_url 수신 (code=0000)
- ✓ HMAC 무결성 검증
- ✓ 결제금액 일치 확인
- ✓ noti_url 수신 (해당 시)
- ✓ 하이브리드 승인 성공 (해당 시)

**단계별 상세**
1. 거래등록 — TID, Trade ID, 요청/응답 JSON
2. ok_url 수신 — code, HMAC 검증, pay_token, sign_date, 전체 파라미터
3. noti_url 수신 — code, HMAC 검증, 전체 파라미터
4. 하이브리드 승인 — 요청/응답 JSON (하이브리드 케이스만)

### 5-4. 실패 원인 분석

실패 시 아래 항목을 자동 분석합니다:

| 실패 유형 | 확인 방법 |
|----------|----------|
| 거래등록 실패 | 에러코드 확인 (6101: 파라미터 오류, 6102: HMAC 오류) |
| ok_url 미수신 | ngrok URL 설정 확인, 방화벽 확인 |
| HMAC 검증 실패 | skey 값 확인, 메시지 구성 순서 확인 |
| 금액 불일치 | 거래등록 요청 금액과 응답 금액 비교 |
| 하이브리드 승인 실패 | pay_token, mc_user_key 등 파라미터 확인 |
| 사용자 취소 | close_url로 리다이렉트됨 |

## 6단계: 이력 관리

### 데이터 저장 위치

모든 테스트 결과는 `auto_test/data/results.json`에 자동 저장됩니다.

### API로 결과 조회

| API | 설명 |
|-----|------|
| `GET /api/results` | 전체 결과 (최신순) |
| `GET /api/results/:caseId` | 특정 케이스 결과만 |
| `GET /api/result/:runId` | 특정 실행 결과 상세 |

### 결과 초기화

테스트 이력을 초기화하려면 `auto_test/data/results.json` 파일을 삭제하면 됩니다.

```
└── 연동테스트_자동화_계획_260404.md
```

## 향후 확장

다른 결제수단 테스트 케이스를 추가할 때:
- `server/test-cases.ts`에 케이스 배열 추가 (RA_CASES, VA_CASES)
- `ALL_CASES`에 포함시키면 대시보드에 자동 표시됨
- `views/index.html`의 `sectionMeta`에 결제수단별 제목/경고 추가

```typescript
// 예: 실계좌이체 케이스 추가
export const RA_CASES: TestCase[] = [
  { id: "RA-01", name: "실계좌이체 기본", cashCode: "RA", ... },
];

export const ALL_CASES: TestCase[] = [...MC_CASES, ...CN_CASES, ...RA_CASES];
```
