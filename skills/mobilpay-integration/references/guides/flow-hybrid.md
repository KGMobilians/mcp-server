# 결제 플로우 - 하이브리드결제

하이브리드결제는 결제 인증과 승인을 분리하여 처리하는 방식입니다. 거래등록 시 `hybrid_pay`를 `Y`로 설정하면 하이브리드결제로 동작합니다. MOBILPAY 결제 창은 인증만 담당하고, 결제 승인은 가맹점이 별도의 승인 API를 호출하여 처리합니다.

## 결제 순서도

```
┌─────────────────────────────────────────────────────────────┐
│ [1] 결제 요청                                                │
│     사용자가 가맹점 사이트에서 결제를 요청합니다.               │
│                                                              │
│ [2] 거래 등록 API 호출                                        │
│     가맹점 서버 → MOBILPAY: POST /MUP/api/registration       │
│     hybrid_pay를 "Y"로 설정하여 거래를 등록합니다.            │
│                                                              │
│ [3] 결제창 호출                                               │
│     거래등록 응답으로 받은 pay_url을 팝업 또는 iframe으로       │
│     호출합니다.                                               │
│                                                              │
│ [4] 결제 진행                                                │
│     사용자는 호출된 결제창에서 결제를 진행합니다.               │
│                                                              │
│ [5] 인증 결과 응답                                            │
│     MOBILPAY → 가맹점: ok_url로 인증 결과를 POST 전달합니다.  │
│     이 시점에서는 아직 승인이 완료되지 않은 상태입니다.         │
│                                                              │
│ [6] 결제 승인 API 호출                                        │
│     가맹점 서버 → MOBILPAY: POST /MUP/api/approval           │
│     인증 결과에서 받은 pay_token 등을 전달하여 승인을 요청합니다.│
│                                                              │
│ [7] 결제 승인 결과 응답                                       │
│     MOBILPAY → 가맹점 서버: 승인 결과를 JSON으로 응답합니다.   │
│                                                              │
│ [8] 서비스 제공                                               │
│     code=0000 확인 후 사용자에게 서비스를 제공합니다.           │
└─────────────────────────────────────────────────────────────┘
```

## 시퀀스 다이어그램

```
사용자          가맹점 서버           MOBILPAY
  │                │                    │
  │─ 결제 요청 ──→│                    │
  │                │── 거래등록 API ──→│  (hybrid_pay=Y)
  │                │←─ pay_url 응답 ──│
  │←─ pay_url ────│                    │
  │── 결제창 열기 ─────────────────→│
  │                │                    │
  │       (사용자 인증 진행)             │
  │                │                    │
  │                │←── 인증결과 POST ──│  ← ok_url (인증만 완료)
  │                │                    │
  │                │── 승인 API 호출 ─→│  ← POST /MUP/api/approval
  │                │←── 승인결과 응답 ──│  (code=0000)
  │                │                    │
  │←─ 결제완료 ──│                    │
  │                │                    │
```

## 장점

- 가맹점이 원하는 시점에 승인 처리가 가능합니다.
- 복합결제, 장바구니 결제 등 결제 관련 프로세스 추가가 용이합니다.
- 인증과 승인이 분리되어 있어, 인증 후 추가 검증 로직을 삽입할 수 있습니다.

## 단점

- 결제 승인을 위한 추가 연동이 필요합니다 (일반결제 대비 개발 범위가 넓음).
- 인증 완료 후 승인 API를 호출하지 않으면 결제가 완료되지 않습니다.

## 일반결제 vs 하이브리드결제 비교

| 구분 | 일반결제 | 하이브리드결제 |
|------|----------|---------------|
| 처리 방식 | 인증+승인 자동 처리 | 인증(결제창) / 승인(API) 분리 |
| 거래등록 설정 | `hybrid_pay=N` (기본값) | `hybrid_pay=Y` |
| ok_url 응답 | 승인 완료 결과 | 인증 완료 결과 |
| 승인 API 호출 | 불필요 | 필요 (`/MUP/api/approval`) |
| 장점 | 구현 간단, 별도 승인 불필요 | 승인 시점 제어, 복합결제 용이 |
| 단점 | 복합결제 추가 어려움, 결과수신 실패 시 불일치 | 개발 범위 넓음, 승인 미호출 시 미완료 |
| 권장 대상 | 단순 결제, 빠른 연동 필요 시 | 복합결제, 장바구니 결제, 승인 시점 제어 필요 시 |

> **주의**: 하이브리드결제와 통합결제(`integrate_pay=Y`)를 동시에 사용하면 실계좌이체(`cash_code=RA`)는 결제수단으로 노출되지 않습니다.
