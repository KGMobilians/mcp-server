# 결제 플로우 - 일반결제

일반결제는 결제 인증과 승인이 동시에 이루어지는 방식입니다. 거래등록 시 `hybrid_pay`를 `N`(기본값)으로 설정하면 일반결제로 동작합니다. 결제 창에서 사용자가 결제를 완료하면 인증과 승인이 자동으로 처리되고, `ok_url`로 승인 결과가 전달됩니다.

## 결제 순서도

```
┌─────────────────────────────────────────────────────────────┐
│ [1] 결제 요청                                                │
│     사용자가 가맹점 사이트에서 결제를 요청합니다.               │
│                                                              │
│ [2] 거래 등록 API 호출                                        │
│     가맹점 서버 → MOBILPAY: POST /MUP/api/registration       │
│     거래등록에 필요한 결제 정보(sid, amount, trade_id 등)를     │
│     전달합니다. hybrid_pay는 "N"(기본값)으로 설정합니다.       │
│                                                              │
│ [3] 결제창 호출                                               │
│     거래등록 응답으로 받은 pay_url을 팝업 또는 iframe으로       │
│     호출합니다.                                               │
│                                                              │
│ [4] 결제 진행                                                │
│     사용자는 호출된 결제창에서 결제수단 선택, 인증, 결제를      │
│     진행합니다.                                               │
│                                                              │
│ [5] 승인 결과 응답                                            │
│     MOBILPAY → 가맹점: ok_url(또는 noti_url)로 승인 결과를     │
│     POST 전달합니다. 인증+승인이 동시에 완료된 상태입니다.      │
│                                                              │
│ [6] 서비스 제공                                               │
│     code=0000 확인, 금액 검증, HMAC 검증을 완료한 후           │
│     사용자에게 서비스를 제공합니다.                             │
└─────────────────────────────────────────────────────────────┘
```

## 시퀀스 다이어그램

```
사용자          가맹점 서버           MOBILPAY
  │                │                    │
  │─ 결제 요청 ──→│                    │
  │                │── 거래등록 API ──→│
  │                │←─ pay_url 응답 ──│
  │←─ pay_url ────│                    │
  │── 결제창 열기 ─────────────────→│
  │                │                    │
  │       (사용자 결제 진행)             │
  │                │                    │
  │                │←── 승인결과 POST ──│  ← ok_url / noti_url
  │                │   (code=0000)      │
  │←─ 결제완료 ──│                    │
  │                │                    │
```

## 장점

- 승인을 위한 별도 처리가 필요하지 않습니다. 결제 창에서 인증과 승인이 동시에 완료됩니다.
- 구현이 간단합니다. 거래등록 → 결제창 호출 → 결과 수신 3단계만 구현하면 됩니다.

## 단점

- 인증과 승인이 동시에 이루어지므로 복합결제, 장바구니 결제 등 결제 관련 프로세스 추가가 용이하지 않습니다.
- 결제 결과 수신 실패 또는 가맹점 결제 처리 실패 시 거래 불일치가 발생할 수 있습니다. `noti_url`을 반드시 활용하세요.

## ok_url 결과 수신 시 필수 처리

1. `code` = `0000` (성공) 확인
2. `amount` = 최초 요청 금액과 동일한지 확인
3. `hmac` 무결성 검증
4. 중복 거래 방어 로직 (이미 처리된 `trade_id`인지 확인)
5. 모든 검증 통과 시 서비스 제공 후 `SUCCESS` 출력
